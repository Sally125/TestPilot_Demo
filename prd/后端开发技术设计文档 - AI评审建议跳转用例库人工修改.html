一、文档概述

需求来源与目标

本设计基于产品需求文档《AI评审建议跳转用例库人工修改》，核心目标是让 AI 评审报告中的每条改进建议能绑定具体的测试用例和字段，用户点击建议后系统自动定向到用例库编辑态，完成修改后建议状态和用例评审状态自动流转。当前系统的评审建议仅以非结构化 JSON 存储在 review_reports.suggestions 字段中，缺少独立的建议实体、状态跟踪和与用例的关联关系，无法支撑"定向跳转-编辑-状态回写"的闭环流程。

技术栈
FastAPI + SQLAlchemy 2.0 + SQLite
变更范围
3 张表 + 6 个接口
优先级
P0 (MVP)

范围与约束

类别内容
本期实现建议结构化存储、状态流转、用例定向查询与保存、重新评审单条用例、忽略建议
本期不做AI一键应用建议（仅预留 sample_patch 字段）、多用户实时协同编辑、建议批量操作
技术约束不修改现有业务代码的核心逻辑，仅做增量扩展；数据库为 SQLite，不引入新的中间件
兼容约束现有 review_reports.suggestions JSON 结构保持向前兼容，前端旧版本仍可读取

术语定义

术语说明
建议 (Suggestion)AI 评审生成的一条改进项，绑定到一个测试用例的一个或多个字段
深链接 (Deep Link)携带 caseId、reviewId、suggestionId、focus 参数的用例库 URL
乐观锁 (Optimistic Lock)通过 version 字段实现的并发控制机制，保存时校验版本号防止覆盖
定向编辑态用例库接收到深链接参数后进入的特殊编辑模式，自动定位用例并高亮目标字段

二、现有架构分析

技术栈与分层

TestPilot 后端基于 FastAPI 0.115.6 框架，使用 SQLAlchemy 2.0.35 作为 ORM，数据库为 SQLite。代码位于 testpilot-backend/app/ 目录下，采用四层分层架构：

所有路由通过 APIRouter(prefix="/api") 注册，使用 Depends(get_db) 注入数据库会话。AI 核心接口声明 response_model 返回 Pydantic 模型，CRUD 接口直接返回 camelCase dict。错误处理在路由函数内通过 try/except 捕获后抛出 HTTPException，RuntimeError 映射为 400，其他异常映射为 500。

现有数据模型

当前与评审和用例相关的 ORM 模型有三张表：

ReviewReport (review_reports)

评审报告表，存储 AI 评审的评分、维度指标和建议。建议以 JSON 数组存储在 suggestions 字段中，每条建议仅包含 case_title、issue、suggestion、example 四个字段，没有独立 ID、没有关联 case_id、没有状态字段。

class ReviewReport(Base):
    __tablename__ = "review_reports"
    id = Column(Integer, primary_key=True, index=True)
    project_id = Column(Integer, nullable=False)
    score = Column(Integer, nullable=True)
    metrics = Column(JSON, nullable=True)
    suggestions = Column(JSON, nullable=True)  # 非结构化建议列表
    req_source = Column(String(500), nullable=True)
    model = Column(String(100), nullable=True)
    reviewed_at = Column(DateTime(timezone=True), server_default=func.now())

TestCase (test_cases)

测试用例表，存储用例标题、模块、优先级、前置条件、操作步骤、预期结果、脚本内容等。有 status 字段（取值 pending、passed、failed），但没有评审状态字段和版本号字段。

Pydantic ReviewSuggestion (models.py)

现有的 ReviewSuggestion 是 Pydantic 模型而非数据库表，仅用于 AI 评审的请求/响应数据结构，不持久化：

class ReviewSuggestion(BaseModel):
    case_title: str = Field(..., description="问题所在用例标题")
    issue: str = Field(..., description="问题描述")
    suggestion: str = Field(..., description="改进建议")
    example: str = Field(default="", description="代码示例（可选）")

现有评审链路

当前评审流程为：POST /api/projects/{project_id}/review/generate 接口从数据库拉取用例和功能点，调用 AIService.review_cases() 方法请求 DeepSeek API，返回包含评分和建议的 ReviewResponse，然后通过 create_review_report() 将结果存入 review_reports 表。建议数据在持久化时被转换为带有 tag、label、title、problem、suggestion、code 字段的 JSON 数组，但仍然缺少建议 ID、用例 ID 关联和状态跟踪。

差距分析

PRD 需求现状差距
每条建议有独立 ID 和状态建议以 JSON 数组存储，无独立实体需新增 review_suggestions 表
建议关联到具体用例 ID仅有 case_title 文本匹配需增加 case_id 外键关系
建议状态流转 (pending/editing/resolved/ignored)无状态字段需增加 status 字段和状态机
用例评审状态 (needs_modification/modified_pending_review/review_passed)用例仅有执行状态 (pending/passed/failed)需增加 review_status 字段
保存时校验版本防并发覆盖无版本控制需增加 version 字段实现乐观锁
保存用例时回写建议状态保存用例与建议无关需在事务中联动更新
重新评审单条用例仅支持整项目评审需新增单用例评审接口
建议包含 field_path、issue_type、severity仅有 problem/suggestion 文本需扩展 AI Prompt 输出结构

三、数据库设计

新增表：review_suggestions

将建议从 review_reports.suggestions JSON 中拆出，作为独立实体存储。每条建议绑定到一个评审报告和一个测试用例，拥有自己的状态、处理人和审计字段。

ORM 模型定义

class ReviewSuggestion(Base):
    """AI 评审改进建议（独立实体）"""
    __tablename__ = "review_suggestions"

    id = Column(Integer, primary_key=True, index=True)
    suggestion_uid = Column(String(20), nullable=False, unique=True, index=True)
    review_report_id = Column(Integer, nullable=False, index=True)
    project_id = Column(Integer, nullable=False, index=True)
    case_id = Column(Integer, nullable=False, index=True)
    field_path = Column(JSON, nullable=False)          # ["steps", "expected_result"]
    issue_type = Column(String(30), nullable=False)     # coverage_gap/boundary_missing/...
    severity = Column(String(10), nullable=False)       # high/medium/low
    problem = Column(Text, nullable=False)
    suggestion = Column(Text, nullable=False)
    sample_patch = Column(JSON, nullable=True)          # AI 参考修改示例
    status = Column(String(20), nullable=False, default="pending")  # pending/editing/resolved/ignored
    handled_by = Column(String(100), nullable=True)
    handled_at = Column(DateTime(timezone=True), nullable=True)
    ignore_reason = Column(Text, nullable=True)
    created_at = Column(DateTime(timezone=True), server_default=func.now())
    updated_at = Column(DateTime(timezone=True), onupdate=func.now())

枚举值定义

字段枚举值说明
issue_typecoverage_gap覆盖缺失：用例未覆盖某功能点或场景
boundary_missing边界缺失：缺少边界值测试

medium中严重度

设计决策：suggestion_uid 使用 S-{review_report_id}-{序号} 格式而非 UUID，原因是建议数量少（每次评审最多 3 条），可读性优于随机 ID。序号三位补零，如 S-5-001。该值在创建建议时由后端生成，不由 AI 输出。

修改表：test_cases

在现有 TestCase 表上增加两个字段，分别用于跟踪评审状态和实现乐观锁。

# 新增字段（追加到现有 TestCase 类）
review_status = Column(String(30), nullable=True)  # 评审状态
version = Column(Integer, nullable=False, default=1)  # 乐观锁版本号
新增字段类型默认值说明
review_statusString(30)NULL用例评审状态：needs_modification / modified_pending_review / review_passed。NULL 表示未评审。
versionInteger1乐观锁版本号，每次保存自增 1

review_status 枚举

值触发条件前端展示
NULL未评审或评审通过后无建议不展示评审状态标签
needs_modificationAI 评审发现问题并生成建议需修改
modified_pending_review用户保存修改后已修改待复审
review_passed重新评审通过已通过

向前兼容：现有用例的 review_status 为 NULL，version 为 1，不影响现有功能。review_status 与现有的 status 字段（执行状态）语义独立，不混用。

修改表：review_reports

在现有 ReviewReport 表上增加一个字段，标记报告是否已过期。

# 新增字段（追加到现有 ReviewReport 类）
is_expired = Column(Integer, nullable=False, default=0)  # 0=有效, 1=已过期

当用户对用例保存修改后，该用例关联的评审报告标记为过期（is_expired=1），前端在报告页顶部展示过期提示。过期报告的建议仍然可查看，但不自动更新状态。

迁移方案

由于项目使用 SQLite 且启动时通过 Base.metadata.create_all(bind=engine) 自动建表，新增表和新增字段需要手动执行迁移脚本。SQLite 不支持 ALTER TABLE ADD COLUMN IF NOT EXISTS，因此使用 try/except 包裹 DDL 语句。

迁移脚本

-- migration_001_review_suggestions.sql

-- 1. 新增 review_suggestions 表
CREATE TABLE IF NOT EXISTS review_suggestions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    suggestion_uid VARCHAR(20) NOT NULL UNIQUE,
    review_report_id INTEGER NOT NULL,
    project_id INTEGER NOT NULL,
    case_id INTEGER NOT NULL,
    field_path JSON NOT NULL,
    issue_type VARCHAR(30) NOT NULL,
    severity VARCHAR(10) NOT NULL,
    problem TEXT NOT NULL,
    suggestion TEXT NOT NULL,
    sample_patch JSON,
    status VARCHAR(20) NOT NULL DEFAULT 'pending',
    handled_by VARCHAR(100),
    handled_at DATETIME,
    ignore_reason TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME
);

CREATE INDEX IF NOT EXISTS idx_suggestions_review_id ON review_suggestions(review_report_id);
CREATE INDEX IF NOT EXISTS idx_suggestions_case_id ON review_suggestions(case_id);
CREATE INDEX IF NOT EXISTS idx_suggestions_status ON review_suggestions(status);
CREATE INDEX IF NOT EXISTS idx_suggestions_project_id ON review_suggestions(project_id);

-- 2. test_cases 新增字段（SQLite 不支持 IF NOT EXISTS，用 Python 迁移）
-- 在 main.py 启动时执行：
--   ALTER TABLE test_cases ADD COLUMN review_status VARCHAR(30);
--   ALTER TABLE test_cases ADD COLUMN version INTEGER NOT NULL DEFAULT 1;

-- 3. review_reports 新增字段
--   ALTER TABLE review_reports ADD COLUMN is_expired INTEGER NOT NULL DEFAULT 0;

Python 迁移代码

# app/migration.py
from sqlalchemy import text
from app.database import engine

def run_migration():
    """启动时执行数据库迁移（幂等）"""
    with engine.connect() as conn:
        # test_cases 新增字段
        for col, ddl in [
            ("review_status", "ALTER TABLE test_cases ADD COLUMN review_status VARCHAR(30)"),
            ("version", "ALTER TABLE test_cases ADD COLUMN version INTEGER NOT NULL DEFAULT 1"),
        ]:
            try:
                conn.execute(text(ddl))
                conn.commit()
            except Exception:
                pass  # 字段已存在

        # review_reports 新增字段
        try:
            conn.execute(text(
                "ALTER TABLE review_reports ADD COLUMN is_expired INTEGER NOT NULL DEFAULT 0"
            ))
            conn.commit()
        except Exception:
            pass

在 main.py 的 Base.metadata.create_all() 之后调用 run_migration()。

四、接口设计

接口总览

新增 6 个 API 接口，遵循现有的 /api 前缀和 RESTful 嵌套路径风格。接口命名与 PRD 定义对齐，但根据现有代码风格做了微调（如路径使用复数资源名）。

接口方法用途优先级
/api/reviews/{reviewId}/suggestionsGET获取评审报告下的建议列表P0
/api/review-suggestions/{suggestionId}GET获取单条建议详情P0
/api/review-suggestions/{suggestionId}/statusPATCH更新建议状态（忽略/恢复）P0
/api/cases/{caseId}GET用例定向查询（含版本号、权限信息）P0
/api/cases/{caseId}PATCH保存用例修改（含建议状态回写）P0
/api/cases/{caseId}/reviewPOST重新评审单条用例P1

命名说明：现有项目中用例查询路径为 /api/projects/{project_id}/testcases，但 PRD 深链接格式为 /cases?caseId=TC-007。为兼容深链接的前端路由，后端新增 /api/cases/{caseId} 独立资源路径，与现有 /api/testcases/{case_id} 路径并存。前者用于定向编辑场景（返回版本号、评审状态、关联建议），后者用于项目内用例管理。

状态流转规则

当前状态允许目标状态说明
pendingignored用户忽略建议
editingignored编辑中途放弃，忽略建议

实现逻辑

  1. 查询用例，不存在返回 404。
  2. 查询用例所属项目的需求分析结果（功能点），不存在返回 400。
  3. 调用 AIService.review_cases() 对单条用例进行评审。

五、业务逻辑设计

状态机设计

建议状态和用例评审状态各自独立流转，但在关键节点（保存修改、重新评审）存在联动。

联动规则

触发事件建议状态变更用例评审状态变更其他副作用
AI 评审完成创建建议，初始 pending→ needs_modification无
用户点击"去修改"pending → editing不变无
用户保存用例修改当前建议 → resolved→ modified_pending_review评审报告标记过期
用户忽略建议→ ignored不变无
用户恢复建议ignored → pending不变无
重新评审通过无新建议→ review_passed无
重新评审仍有问题创建新建议，初始 pending→ needs_modification无

保存用例的事务流程

保存用例修改是整个需求中最复杂的操作，涉及多表联动。以下是事务内部的执行步骤：

CRUD 层实现

# crud.py 新增函数

def update_case_with_suggestion(
    db: Session,
    case_id: int,
    update_fields: dict,
    expected_version: int,
    suggestion_uid: str | None = None,
    handled_by: str | None = None,
) -> tuple[TestCase | None, str | None]:
    """
    事务性更新用例并联动建议状态。
    返回 (更新后的用例, 错误信息)。
    错误信息为 None 表示成功。
    """
    case = db.query(TestCase).filter(TestCase.id == case_id).with_for_update().first()
    if not case:
        return None, "CASE_NOT_FOUND"

    # 乐观锁校验
    if case.version != expected_version:
        return None, "VERSION_CONFLICT"

    try:
        # 1. 更新用例字段
        for key, value in update_fields.items():
            if hasattr(case, key) and value is not None:
                setattr(case, key, value)

        # 2. 版本号自增
        case.version += 1

        # 3. 评审状态流转
        case.review_status = "modified_pending_review"

        # 4. 建议状态回写
        if suggestion_uid:
            suggestion = db.query(ReviewSuggestion).filter(
                ReviewSuggestion.suggestion_uid == suggestion_uid
            ).first()
            if suggestion:
                suggestion.status = "resolved"
                suggestion.handled_by = handled_by
                suggestion.handled_at = datetime.datetime.now(datetime.timezone.utc)

                # 5. 标记评审报告过期
                db.query(ReviewReport).filter(
                    ReviewReport.id == suggestion.review_report_id
                ).update({ReviewReport.is_expired: 1})

        db.commit()
        db.refresh(case)
        return case, None

    except Exception as e:
        db.rollback()
        return None, f"TRANSACTION_ERROR: {e}"

乐观锁与并发控制

SQLite 默认使用 DEFERRED 事务隔离级别，写操作会获取排他锁。结合 version 字段实现乐观锁，核心逻辑是：读取时获取当前 version，保存时校验 version 是否变化，若未变则更新并自增，若已变则拒绝保存。

场景客户端 A客户端 B结果
正常保存读取 version=1-保存成功，version→2
顺序保存读取 v=1，保存成功 v→2读取 v=2，保存成功 v→3两次都成功
并发冲突读取 v=1读取 v=1A 先保存 v→2，B 保存时 v≠1 返回 409

由于 SQLite 不支持 SELECT ... FOR UPDATE 的行级锁（with_for_update() 在 SQLite 中会被忽略），乐观锁是主要的并发控制手段。在高并发场景下（本项目为单用户系统，实际不会出现），可考虑迁移到 PostgreSQL 使用行级锁。

AI评审Prompt改造

现有 REVIEW_PROMPT 输出的建议仅包含 case_title、issue、suggestion、example 四个字段。需改造 Prompt，让 AI 输出结构化的建议数据，包含 case_id、field_path、issue_type、severity、sample_patch 等字段。

改造后的 Prompt 输出格式

"suggestions": [
  {
    "case_title": "用例标题",
    "case_index": 0,
    "field_path": ["steps", "expected_result"],
    "issue_type": "coverage_gap",
    "severity": "medium",

  • 新增 field_path 数组，限定取值范围：title、precondition、steps、expected、script。
  • 新增 issue_type 和 severity 枚举，在 Prompt 中给出明确的取值列表和定义。
  • sample_patch 为结构化 JSON，按 field_path 中的字段给出参考修改内容。
  • 保留 example 字段用于代码示例，向前兼容。

@router.patch("/cases/{case_id}")
async def update_case(
    case_id: int,
    body: CaseUpdateRequest,
    db: Session = Depends(get_db),
):
    case = get_test_case(db, case_id)
    if not case:
        raise HTTPException(
            status_code=404,
            detail="该用例已不存在，可能已被删除",
            headers={"X-Error-Code": "CASE_NOT_FOUND"},
        )

    updated, error = update_case_with_suggestion(
        db, case_id,
        update_fields={
            "title": body.title,
            "precondition": body.precondition,
            "steps": body.steps,
            "expected": body.expected,
            "script": body.script,
        },
        expected_version=body.version,
        suggestion_uid=body.suggestion_id,
    )

    if error == "CASE_NOT_FOUND":
        raise HTTPException(status_code=404, detail="该用例已不存在")
    if error == "VERSION_CONFLICT":
        raise HTTPException(
            status_code=409,
            detail="用例版本冲突，请刷新后重试",
            headers={"X-Error-Code": "VERSION_CONFLICT",
                     "X-Server-Version": str(case.version)},
        )
    if error:
        raise HTTPException(status_code=500, detail=f"保存失败: {error}")

    return {
        "id": updated.id,
        "version": updated.version,
        "reviewStatus": updated.review_status,
        "suggestionStatus": "resolved" if body.suggestion_id else None,
        "message": "已保存，建议状态已更新",
        "canRecheck": True,
    }

风险与对策

风险影响概率对策
AI 输出的 case_index 与实际用例不匹配建议关联到错误用例中后端校验 case_index 范围，越界时跳过该建议并记录日志
AI 输出的 field_path 不在合法枚举内前端无法高亮字段低后端过滤非法值，仅保留合法字段名
SQLite 并发写入锁冲突保存操作偶发失败低设置重试机制（3次，间隔 200ms），仍失败返回 503
现有 review_reports.suggestions JSON 格式变更前端旧版本解析失败中保持 JSON 字段向前兼容，新增字段为可选
DeepSeek API 超时（单用例评审）重新评审无响应中设置 60s 超时，超时后返回 504 并提示用户重试
迁移脚本在已有数据的环境执行失败启动报错低迁移脚本使用 try/except 幂等设计，失败时记录日志但不阻断启动
